## <b>📝 형상 관리 (Configuration Management)</b>
 : 소프트웨어 개발 과정의 변경 사항 관리

### <b>📌 형상 관리 절차 ⇒ <b><span style='color:#2D3748; background-color:#fff5b1'>식통감기</span></b></b>

- 형상 식별
- 형상 통제
  - 형상통제위원회(CBB)가 형상 항목의 버전 관리를 위해 변경 여부와 변경 활동 통제
- 형상 감사
  - 소프트웨어 베이스라인 변경 시 무결성, 요구사항과의 일치 여부 평가
- 형상 기록

### <b>📌 형상 관리 도구 유형 ⇒ <b><span style='color:#2D3748; background-color:#fff5b1'>공클분</span></b></b>

- 공유 폴더 방식 (RCS, SCCS)
  - 개발 완료된 파일 : 공유 폴더에 복사
  - 담당자가 매일 공유 폴더를 자기 PC로 복사 + 컴파일 → 에러 확인
  - 정상일 경우 다음날 각 개발자들이 동작 여부 확인
- 클라이언트/서버 방식 (CVS, SVN)
  - 중앙 버전 관리 시스템 항시 동작
  - 개발자들의 작업 내용 추적 굿
  - 서로 다른 개발자가 같은 파일 작업 시 경고 메시지 출력
- 분산 저장소 방식 (Git)
  - 로컬 저장소와 원격 저장소로 분리 분산 저장
  - 중앙 저장소에서 로컬 파일 복사(Clone) → 로컬 저장소 생성
  - 개발 후 로컬에서 우선 Commit → 원격 저장소에 반영(Push)

# <b>📂 공통 모듈 구현</b>

- 모듈 : 그 자체로 하나의 기능을 수행할 수 있는 독립된 실체
- 모듈화 : 기능 단위 모듈로 분해
- 공통 모듈 : 전체 프로그램 기능 중 특정 기능을 처리하는 실행 코드

## <b>📝 모듈 응집도 ⇒ <b><span style='color:#2D3748; background-color:#fff5b1'>우논시절 통순기</span></b></b>
 : 모듈 내부 구성요소 간 연관 정도 → 하나의 모듈은 하나의 기능을 수행할수록 응집도가 높아짐

- 우연적 응집도 (Coincidental Cohesion)
 : 요소 간 연관이 없는 경우
- 논리적 응집도 (Logical Cohesion)
 : 비슷한 성격, 형태로 분류되는 요소가 한 모듈에서 처리되는 경우
- 시간적 응집도 (Temporal Cohesion)
 : 특정 시간에 처리되는 활동을 한 모듈에서 처리하는 경우
- 절차적 응집도 (Procedural Cohesion)
 : 모듈 요소가 순차적으로 수행될 경우
- 통신적 응집도 (Communication Cohesion)
 : 동일한 입출력으로 다른 기능을 수행하는 활동이 모여있는 경우
- 순차적 응집도 (Sequential Cohesion)
 : 모듈 내 한 활동의 출력값을 다른 활동이 사용할 경우
- 기능적 응집도 (Functional Cohesion)
 : 모듈 내부의 모든 기능이 단일 목적으로 수행되는 경우

<br>

## <b>📝 모듈 결합도 ⇒ <b><span style='color:#2D3748; background-color:#fff5b1'>내공 외제 스자</span></b></b>
 : 모듈 간 관련성, 상호의존성 측정

- 내용 결합도 (Content Coupling)
 : 다른 모듈의 변수, 기능을 다른 모듈에서 직접 참조
- 공통 결합도 (Common Coupling)
 : 모듈 밖의 공통 데이터(전역변수)를 여러 모듈이 사용
- 외부 결합도 (External Coupling)
 : 외부의 데이터포맷, 통신 프로토콜 등을 공유하는 경우
- 제어 결합도 (Control Coupling)
 : 다른 모듈의 내부 논리 조직을 제어하기 위해 제어 신호로 통신하는 경우 → 하위 모듈에서 상위 모듈로 제어 신호 이동, 상위 모듈에게 처리 명령을 부여하는 권리 전도 현상 발생
- 스탬프 결합도 (Stamp Coupling)
 : 동일한 자료 구조를 조회하는 경우 (자료 구조의 변화는 모든 모듈에 영향)
- 자료 결합도 (Data Coupling)
 : 모듈 간 파라미터로만 상호 작용이 일어나는 경우 (한 모듈 내용을 변경해도 다른 모듈에 영향X)

<br>

## <b>📝 공통 모듈 구현 절차 - MVC 패턴</b>
 : UI로부터 비즈니스 로직을 분리해 서로 영향 없이 쉽게 고칠 수 있는 패턴

- Model
  - 애플리케이션이 무엇을 할 것인지
  - 내부 비즈니스 로직 처리
- View
  - 화면에 보여주기
  - 모델, 컨트롤러가 보여주려 하는 것들 처리
- Conrtoller
  - 모델이 어떻게 처리할지
  - 뷰에 명령을 보내 화면 요청 결과 전달

<br>

## <b>📝 Fan-In, Fan-Out</b>
 : 시스템 복잡도 측정 지표

- Fan-In
  - 모듈을 호출하는 모듈 수 (들어오면 in)
  - 팬인이 높으면 : 재사용성 굿, but 단일 장애점 발생 가능, 관리 비용 증가
- Fan-Out
  - 모듈에 의해 호출되는 모듈 수 (나가면 out)
  - 팬아웃 높으면 : 단순화 필요 (불필요한 모듈 호출 여부 검토)
- 시스템 복잡도를 최적화하기 위해서는 팬인은 높게, 팬아웃은 낮게

<br>

## <b>📝 공통 모듈 테스트</b>
 : IDE 도구로 개별 공통 모듈 디버깅
- 공통 모듈 테스트 : 화이트박스 기법
- 단위 테스트 : jUnit

### <b>📌 공통 모듈 테스트</b>

- 화이트박스 테스트
 : 프로그램 소스 코드로 구조, 동작 검사
- 메서드 기반 테스트
 : 공통 모듈 외부에 공개된 메서드 기반 테스트 (파라미터 조정하며 테스트)
- 화면 기반 테스트
 : 화면단위로 단위모듈 개발 후 직접 데이터 입력하며 테스트
- 테스트 드라이버/스텁 활용
 : 드라이버(하위 모듈은 있지만 상위 모듈 없는 경우), 스텁(상위 모듈은 있지만 하위 모듈이 없는 경우)

### <b>📌 jUnit</b>
 : 자바 프로그래밍 언어용 단위 테스트 도구

@Test : 테스트 메서드 선언
@Before, @After : 테스트 전후 실행되는 코드
@BeforeClass, @AfterClass : 테스트 앞뒤로 한 번 수행
@Ignore : 테스트에서 제외할 메서드

# <b>📂 서버 프로그램 구현</b>

1. DTO, VO 구현
   - DTO (Data Transfer Object) : 데이터 전송 객체 (작은 가변 클래스 객체)
   - VO (Value Object) : 고정 클래스를 가지는 객체
2. SQL문, MyBatis XML 구현
   - MyBatis : 객체 지향 언어인 자바의 관계형 DB 프로그래밍을 돕는 개발 프레임워크
3. DAO 구현 (Data Access Object)
   - 특정 타입 데이터베이스에 추상 인터페이스 제공하는 객체
   - 세부내용 노출 없이 데이터 조작하는 객체
4. Service 구현
5. Controller 구현
6. 입/출력 검증 로직 구현

# <b>📂 배치 프로그램 구현</b>

## <b>📝 배치 프로그램 유형 ⇒ <b><span style='color:#2D3748; background-color:#fff5b1'>이온정</span></b></b>

- 이벤트 배치
 : 조건 충족 시 자동 실행
- 온디맨드 배치
 : 사용자의 명시적 요구 있을 때마다 실행
- 정기 배치
 : 정해진 시점에 실행

## <b>📝 배치 스케줄러</b>
 : 일괄처리(Batch Programming)를 위해 주기적으로 발생하는 작업 지원

- 스프링 배치 (Spring Batch)
 : 스프링 프레임워크의 스케줄러 배치 애플리케이션
- 쿼츠 스케줄러 (Quartz Scheduler)
 : 스프링 프레임워크에 플러그인되어 작업과 트리거를 분리해 유연성 제공

## <b>📝 Cron 표현식 (쿼츠 크론 표현식 기준)</b>
 : 스케줄러 실행을 위해 Cron 표현식으로 배치 시간 설정

- 초 - 분 - 시 - 일 - 월 - 요일 - 연도(생략가능)
- Cron 표현식 특수문자
  - \* : 모든 수
  - ? : 해당 항목 사용X
  - \- : 기간설정
  - , : 특정 기간 설정
  - / : 시작시간, 반복간격
  - L : 마지막 기간에 동작
  - W : 가장 가까운 평일에 동작
  - \# : 몇 번째 주, 요일 설정
